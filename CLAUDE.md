# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A reading club management app ("Sonhos Literários", pt-BR UX throughout): Django REST Framework backend (`backend/`) + React/TypeScript frontend (`frontend/`, rebranded from the TailAdmin dashboard template — most demo/template pages have been removed). Domains: member accounts/invites, author/publisher/book catalog (with ISBN lookup via Open Library), reading/meeting tracking, a public blog, and subscription billing with manual (PIX/cash) confirmation plus a Mercado Pago Checkout Pro integration for automatic confirmation via webhook.

## Commands

### Backend (`backend/`)
```bash
source venv/bin/activate            # virtualenv already present in repo
python manage.py runserver          # http://127.0.0.1:8000
python manage.py migrate
python manage.py makemigrations
python manage.py test               # e.g. python manage.py test club.tests
python manage.py renew_subscriptions  # generates advance payments, emails/notifies, expires unpaid subs
python manage.py send_meet_reminders  # emails/notifies participants of meets happening tomorrow
```
Both management commands now run on their own daily via `billing/scheduler.py` (APScheduler, started from `BillingConfig.ready()`) — no external cron needed for local/single-process use; see the caveat below.

Test coverage is now solid across every app: `billing/tests.py` (payment confirmation, pending queue, plan management, Mercado Pago preference/webhook — all mocked, no real MP credentials — plus `renew_subscriptions`), `club/tests.py` (notification mark-seen, ISBN lookup, public stats, `send_meet_reminders`), `accounts/tests.py` (register, password reset, login rate limiting, profile/password/member/invite-code views), and `api/tests.py` (every permission class in `api/permissions.py`, plus the analytics endpoint).

Env vars (all read via `python-dotenv` from `backend/.env`, see `.env.example`):
- `SECRET_KEY` — falls back to the original dev key if unset, so local setups work without a `.env`; production must set a real one.
- `MERCADOPAGO_ACCESS_TOKEN`/`MERCADOPAGO_PUBLIC_KEY`/`MERCADOPAGO_WEBHOOK_SECRET` — no test credentials exist yet, so the integration is unit-tested only, never exercised against the real API.
- `EMAIL_HOST_USER`/`EMAIL_HOST_PASSWORD` (+ `EMAIL_HOST`/`EMAIL_PORT`/`EMAIL_USE_TLS`, defaulting to Mailgun's SMTP relay) — switches `EMAIL_BACKEND` from the console backend to real SMTP once both are set; no Mailgun credentials configured yet either.

### Frontend (`frontend/`)
```bash
npm run dev        # vite dev server, http://localhost:5173, proxies /api and /media to :8000
npm run build       # tsc -b && vite build
npm run lint
npm run generate-api  # regenerates frontend/src/api/generated from ../backend/schema.yaml (openapi-typescript-codegen)
```
Run `python manage.py spectacular --file schema.yaml` in `backend/` before `generate-api` if backend serializers/views changed, since the generator reads the committed `backend/schema.yaml`, not a live server.

## Backend architecture

Django project `reading_club` with app-per-domain layout, all mounted under versioned API routes:

- `reading_club/urls.py` → `/api/` → `api/urls.py` → `/v1/` → `api/v1/urls.py`, which includes each domain's own `urls.py`:
  - `accounts/` — custom email-based `User` model (`AUTH_USER_MODEL = accounts.User`, no username), `Profile` (auto-created via post_save signal in `accounts/signals.py`), `InviteCode` for gated registration. Auth uses `djangorestframework-simplejwt` (`/auth/login/` is `ThrottledTokenObtainPairView`, rate-limited via `api.throttling.LoginRateThrottle` — 5/min per IP — to slow down credential brute-forcing; `/auth/refresh/`).
  - `club/` — core domain: `Author`/`Publisher`/`Book` (`BookViewSet.lookup_isbn` action fetches metadata + cover from the Open Library API, admin-only), `Reading` (+ `ReadingUser` through-table, `ReadingStatus` choices), `Meet`/`MeetUser`/`MeetPhoto`, `Notification` (`NotificationViewSet.mark_seen` action; a generic FK — `content_type`/`object_id`/`content_object` — optionally links a notification to the `Payment`/`Subscription`/`Meet` it's about, exposed to the frontend as `target_type`/`target_id`; the dropdown builds a deep link from that when present — e.g. a `meet` target navigates to `/meets?highlight=<id>`, which scrolls to and rings that row — falling back to the static `NOTIFICATION_ROUTES` map in `NotificationDropdown.tsx` by `type` otherwise), a public `BlogPost`/`BlogCategory`, `Quote` (short pull-quotes for the landing page's "Vozes do clube" carousel — admin CRUD via `QuotesPage.tsx`/`QuoteViewSet`, public reads only `is_active=True`, same admin-writes/public-reads `get_permissions()` split as `PublicBlogPostViewSet`), `ContactMessage` (public landing-page contact form submissions — anyone can `POST /club/contact-messages/`, throttled via `ContactMessageRateThrottle`; only admins can list/read/mark-read/delete, via `ContactMessagesPage.tsx`; `perform_create` emails `settings.CLUB_CONTACT_EMAIL` through `api/emails.py`'s `send_notification_email`), `TeamMember`/`TimelineEntry` (the landing page's "Membros"/"História" content — admin CRUD with photo upload via `TeamMembersPage.tsx`/`TimelineEntriesPage.tsx`, `parser_classes = [MultiPartParser, FormParser]` on both ViewSets same as `PublicBlogPostViewSet`; public reads only `is_active=True`; both were seeded from the real photos/copy that used to be hardcoded in the frontend via a data migration that copies files from `frontend/public/images/landing/{team,timeline}/` into `MEDIA_ROOT`), and `PublicClubStatsView` (`AllowAny` `GET /club/public-stats/`, powers the landing page's counters — `reading_hours` has no tracked source field, so it's estimated from pages read at a fixed pace, see `READING_PAGES_PER_HOUR`). `club/management/commands/send_meet_reminders.py` notifies (in-app + email) every participant of a meet happening tomorrow, idempotent per meet/user via that same generic FK.
  - `billing/` — `Plan` (`is_default` marks the plan auto-assigned to new registrants; `PlanWriteSerializer` enforces only one default at a time), `Subscription` (`SubscriptionStatus`; `custom_price` optionally overrides the plan's price for that member permanently, `surcharge_amount`/`surcharge_reason` add a one-off amount to just the next generated payment and then clear themselves — `effective_base_price` is `custom_price or plan.price`), `Payment` (`PaymentMethod`, `PaymentStatus`). Business rules live in `Payment.confirm()` and `billing/signals.py`: a `pre_save`/`post_save` pair on `Payment` detects the transition into `CONFIRMED` and renews the subscription + creates a `Notification`; another signal auto-creates a `PENDING` `Subscription` on the default `Plan` for every new non-staff `User`. `PlanViewSet.destroy()` catches the `Subscription.plan` `PROTECT` constraint and returns a friendly 409 instead of a raw 500. `SubscriptionAdminViewSet` (`GET`/`PATCH` `/billing/subscriptions/`, admin-only via `IsAdmin` — not `IsFinancial`, since assigning plans/pricing is an admin decision) lets an admin list every member's subscription and change its `plan`/`custom_price`/`surcharge_amount`/`surcharge_reason`; `status`/dates stay read-only there (only sinal/`renew_subscriptions` touch them), and a plan/price change only affects the *next* payment `renew_subscriptions` generates (`Payment.amount` is a snapshot taken at generation time, so already-pending payments are unaffected) — exposed in `pages/Members/MembersPage.tsx` as extra columns + an "Editar assinatura" modal per member, alongside the existing invite-code management. `billing/mercadopago_service.py` wraps the `mercadopago` SDK (`create_payment_preference`, `fetch_mp_payment`, `verify_webhook_signature` — HMAC over Mercado Pago's `x-signature`/`x-request-id` manifest); `MercadoPagoPreferenceView` (member, own pending payment only) and `MercadoPagoWebhookView` (`AllowAny`, always re-fetches the payment from Mercado Pago's API rather than trusting the notification body, idempotent against duplicate deliveries) in `billing/views.py` call into it and ultimately reuse `Payment.confirm()` so an automatic MP confirmation renews a subscription exactly like a manual one. `billing/scheduler.py` starts an in-process APScheduler (from `BillingConfig.ready()`, guarded against double-starting under `runserver`'s autoreloader and skipped for one-off management commands) that runs `renew_subscriptions` and `send_meet_reminders` daily — **caveat: in-process, so a multi-worker WSGI deployment needs an external scheduler instead** (system cron or Celery beat calling the same commands).
  - `api/` — cross-cutting: `permissions.py` (`IsAdmin`, `IsFinancial`, `IsOwner`, `IsAdminOrReadOnly`, `IsMemberWithActiveSubscription`, `IsNotificationOwner`, composable via DRF's `|`/`&` — e.g. `IsAdmin | IsFinancial`) used across the other apps' views; `throttling.py` (`LoginRateThrottle`); `emails.py` (`send_notification_email`, logs instead of raising on failure — used by the scheduled commands so one bad address doesn't abort a batch); `views.py` (`AnalyticsView`, `GET /api/v1/analytics/`, admin-or-financial — revenue by month, payments by method, subscription/reading status breakdowns, member growth, meets by month, and a KPI summary including MRR, all aggregated from real data, no models of its own).
- API docs are served via `drf-spectacular` at `/api/schema/`, `/api/docs/`, `/api/redoc/`; `SCHEMA_PATH_PREFIX` strips the `/api/v[0-9]` prefix from the generated schema.
- DRF defaults (in `reading_club/settings.py`): JWT auth, `IsAuthenticated` by default, URL path versioning (`v1` only), `DEFAULT_THROTTLE_RATES` (currently just `"login"`).
- Media uploads (`profiles/`, `books/covers/`, `meetings/`, `blog/`, `receipts/`) go to `backend/media/`, served by Django only when `DEBUG=True`.

## Frontend architecture

- Routing in `src/App.tsx` via `react-router` v7 (`BrowserRouter`). `/` is the public marketing landing page (`pages/Landing/LandingPage.tsx`, not wrapped in `ProtectedRoute`) — deliberately accessible to everyone regardless of auth state (logged-out visitors, members, and admins all see the same page); its header/hero CTA swaps between "Área de Membros" → `/signin` and "Ir para o Painel" → `/dashboard` based on whether `useAuth()` returns a `user`, rather than redirecting away. The authenticated app lives at `/dashboard` and beyond, nested under a single `<ProtectedRoute><AppLayout /></ProtectedRoute>` route, with `/signin` and `/signup` outside it. `ProtectedRoute` (`src/components/ProtectedRoute.tsx`) reads `AuthContext` and redirects to `/signin` when there's no user.
- Two parallel ways to call the backend — know which one a file uses before editing it:
  - `src/api/client.ts` — hand-written `apiRequest<T>()` fetch wrapper used by most feature pages (Books, Readings, Billing, auth forms, profile). Reads/writes JWTs in `localStorage` (`access`/`refresh`), auto-refreshes on a 401 via `/auth/refresh`, force-redirects to `/signin` if refresh fails, and surfaces failures through the toast system (`utils/toast.ts`, `utils/apiErrors.ts`).
  - `src/api/generated/` — OpenAPI client generated by `npm run generate-api` from `backend/schema.yaml`; configured in `main.tsx` (`OpenAPI.BASE`, `OpenAPI.TOKEN`). Regenerate rather than hand-edit anything under this directory.
  - `src/api/config.ts` holds `API_HOST`/`API_PREFIX` (`VITE_API_HOST` env var, default `http://127.0.0.1:8000`, prefix `/api/v1`).
- `AuthContext` (`src/context/AuthContext.tsx`) owns the current `user`, `login`/`logout`, and `loading` state; it's the source of truth `ProtectedRoute` and header/profile components read from.
- The demo/template-only TailAdmin surfaces (`Charts`, `UiElements`, `Tables`, `Forms`, `Calendar`, ecommerce widgets, the sidebar's "Sistema" section) have been removed — the product surfaces are `pages/Books`, `pages/Catalog` (authors/publishers, admin-only), `pages/Readings`, `pages/Meets`, `pages/Blog`, `pages/Billing` (+ `Plans`, `Quotes`, `ContactMessages`, `TeamMembers`, `TimelineEntries` and `Members`, all admin-only, `PaymentConfirmationsPage`, financial-only, and `Analytics`, admin-or-financial — charts via `react-apexcharts`, already a dependency), `pages/AuthPages`, and `pages/UserProfiles`. Admin/financial-only routes are gated both in `AppSidebar.tsx` and in the page itself (`<Navigate>` redirect if the relevant `user.is_admin`/`is_financial` check fails). `pages/Quotes/QuotesPage.tsx` and `pages/ContactMessages/ContactMessagesPage.tsx` both follow `PlansPage.tsx`'s exact CRUD pattern (hand-written `apiRequest`, two separate `Modal`s for create/edit vs. delete-confirm); `pages/TeamMembers/TeamMembersPage.tsx` and `pages/TimelineEntries/TimelineEntriesPage.tsx` add photo upload on top of that same pattern (dropzone-style `<input type="file">` + preview + `FormData`/`apiRequest(..., fd)` submission, copied from `BooksPage.tsx`'s book-cover upload).
- **Operate-mode design system** (see `DESIGN.md` for the full rationale): `AppSidebar.tsx`'s `navGroups` is an array of `{ label, items }` (not a flat `navItems` list) rendering under headers — Painel (ungrouped) · Clube de Leitura · Minha Conta · Financeiro · Site Público · Administração — with the same `admin`/`financial`/`adminOrFinancial` filtering as before, now applied per-group (`canSee()`); a group with zero visible items renders no header. Submenu open/close state is keyed by item name (string), not array index, since items now live inside groups. Three fonts split by job: `font-heading` (Playfair Display, titles), `font-body` (Libre Baskerville, editorial prose only — blog posts, pull-quotes), `font-ui` (Inter, new — the workhorse for nav/tables/forms/buttons/badges, since serif text fights legibility at data density). Shared components in `components/ui/card/` (`Card`, `StatCard`), `components/common/` (`PageHeader`, `EmptyState`) and `components/ui/table/` (`Table`/`TableHeader`/`TableBody`/`TableRow`/`Th`/`Td`) replace the hand-rolled `rounded-2xl border ...` card/table recipe that used to be repeated per page — new Operate-mode pages should use these rather than re-inlining the classes. `AppSidebar.tsx`/`UserDropdown.tsx` are fully pt-BR now (the profile dropdown dropped its dead "Account settings"/"Support" entries, which both pointed at `/profile`, and its role label covers admin/financial/member). `AuthPageLayout.tsx`'s decorative panel uses the brand guide's own hero photograph (`public/images/landing/hero-bg.jpg`) with a `brand-900/700` duotone overlay instead of the old generic `GridShape` dot-grid (deleted, no other callers).
- Brand identity (colors, logo, typography) lives in `src/index.css`'s `@theme` block (`brand`/`error` Tailwind color ramps derived from the club's guide) and `public/images/logo/`; there's no `tailwind.config.js` (Tailwind v4).
- The landing page's copy/photos (`public/images/landing/`) were carried over from the club's old static site (kept at `landingpage/` in the repo root, its own nested git repo, gitignored as reference-only — not part of the build).
- Vite dev server proxies `/api` and `/media` to `http://127.0.0.1:8000` (`vite.config.ts`), so the backend must be running for the frontend to function locally.
- The landing page (`pages/Landing/`) has its own editorial motion stack, isolated from the rest of the app: `gsap`/`ScrollTrigger` and `lenis` (smooth/momentum scroll) are dynamically imported only on `/` via `hooks/useLandingScroll.ts` (mounted once by `LandingPage.tsx`, torn down on unmount so nothing leaks into the authenticated app's native scroll) and `lib/landingMotion.ts` (caches the gsap/ScrollTrigger dynamic import, plus `prefersReducedMotion()`/`supportsFineHover()` helpers). `context/LandingScrollContext.tsx` exposes the active Lenis instance's `scrollTo` to child components (header/hero anchor links). Under `prefers-reduced-motion`, Lenis/GSAP are never imported at all — every section instead falls back to the pre-existing one-shot `hooks/useScrollReveal.ts` (opacity/translateY, zero-cost in that branch). `components/landing/DistortImage.tsx` is a `<img>` drop-in that adds a subtle Three.js hover-distortion shader (`three` is dynamically imported per-instance, only once a thumbnail nears the viewport, and only on non-touch/fine-hover devices) — used on `HistoryTimeline.tsx`'s and `TeamSection.tsx`'s photos only, not page-wide; its texture is deliberately left untagged (`NoColorSpace`, not `SRGBColorSpace`) since the custom `ShaderMaterial` has no Three.js color-management shader chunks — tagging it sRGB double-transforms the color and renders visibly darker/over-saturated than the plain `<img>` it replaces. A landing-only warm-neutral `stone-*` color ramp lives alongside `gray-*`/`brand-*` in `index.css`'s `@theme` block — `stone-*` is for landing components only, the rest of the app (including the `AppHeader`/`AppSidebar` chrome, which got a lighter typography/shadow brand-consistency pass with no new motion) keeps using `gray-*`/`brand-*`. `VozesDoClube.tsx` renders the `Quote` model (see below) as a `swiper/react` carousel (`effect="fade"` — a horizontal "slide" effect left a hairline sub-pixel bleed of the adjacent off-screen slide's text at the clip boundary in Chromium, not worth chasing further once fade sidestepped it entirely) — the CSS pagination/nav styling (`.quotes-carousel` in `index.css`) needs the compound selector `.quotes-carousel.swiper` to out-specificity `swiper-bundle.css`'s own `.swiper { padding: 0 }`, since that stylesheet is imported in `main.tsx` after `index.css` and wins same-specificity ties by source order otherwise. `ClosingCta.tsx` is the landing page's public contact form (posts to `ContactMessage`, see below), not a signup CTA — membership sign-in links live in the header/hero instead.

## graphify

This project has a knowledge graph at graphify-out/ with god nodes, community structure, and cross-file relationships. It is the **required starting point** for understanding this codebase — do not scan/grep the whole project as a first step.

Rules:
- **Starting any analysis or investigation**: always run `graphify query "<question>"` first (use `graphify path "<A>" "<B>"` for relationships, `graphify explain "<concept>"` for focused concepts) before reading raw files or grepping broadly. These return a scoped subgraph, usually much smaller than GRAPH_REPORT.md or raw grep output. Only fall back to reading files directly, or to `GRAPH_REPORT.md` for broad architecture review, when query/path/explain don't surface enough context.
- If graphify-out/wiki/index.md exists, use it for broad navigation instead of raw source browsing.
- **After making ANY code change** (every edit, every commit, not just at the end of a large task): run `graphify update .` before ending the turn. This is mandatory, not opportunistic — the graph must never be left stale relative to the working tree.
- After any change that affects what's documented above (new/removed app, model, route, API client pattern, architectural decision), update the relevant section of this CLAUDE.md in the same turn — don't let it drift out of sync with `graphify update .`. Keep both refreshed together, not one without the other.
